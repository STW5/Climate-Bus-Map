# Plan: Climate Bus Map

> 기후동행카드 사용 가능 버스를 지도 기반으로 쉽게 찾는 서비스

- **작성일**: 2026-03-04
- **레벨**: Dynamic
- **상태**: Plan 완료

---

## 1. 문제 정의

### 배경

서울의 기후동행카드는 특정 버스 노선과 지하철에서만 사용 가능하다.
그러나 이용자는 **어떤 버스가 기후동행카드로 이용 가능한지 직관적으로 확인하기 어렵다.**

### 핵심 문제

- 주변 정류장에 오는 버스 중 기후동행카드 사용 가능 여부를 한눈에 알 수 없다
- 기존 서울버스 앱이나 지도 앱은 기후동행 필터 기능을 제공하지 않는다

### 해결 목표

> 지도 기반으로 기후동행카드 사용 가능한 버스를 쉽게 찾을 수 있는 서비스 제공

---

## 2. 핵심 기능 (Core Features)

### F-01. 주변 정류장 조회

- 사용자 현재 위치 기반으로 반경 500m 이내 버스 정류장 표시
- 지도 위에 정류장 마커로 시각화

### F-02. 버스 도착 정보 조회

- 정류장 선택 시 도착 예정 버스 목록 표시
- 예: `402번 2분 / 201번 5분 / 721번 8분`

### F-03. 기후동행 사용 가능 여부 표시

- 각 버스 노선별 기후동행카드 사용 가능 여부 표시
- 예: `402 🟢 사용 가능 / 201 🔴 사용 불가`

### F-04. 지도 기반 UI

- 현재 위치, 주변 정류장, 버스 정보를 지도 위에 통합 표시
- Kakao Map API 사용

---

## 3. 차별화 기능 (Differentiators)

### D-01. 기후동행 가능 버스 필터

- 지도에서 기후동행 사용 가능한 버스/정류장만 필터링해서 표시

### D-02. 주변 이용 가능 버스 추천

- 현재 위치 기준 이용 가능한 기후동행 버스 노선 목록 제공
- 예: `내 주변 이용 가능 노선: 402, 721, 201`

### D-03. 기후동행 최적 경로 탐색

- 일반 경로 vs 기후동행 경로 비교 제공
- 기후동행 가능한 버스만 사용하는 경로 탐색

### D-04. 버스 도착 알림

- 특정 정류장에서 기후동행 가능한 버스 도착 시 알림

## 4. 기술 스택

| 구분 | 기술 | 역할 |
|------|------|------|
| Backend | Spring Boot, Spring Web, Spring Data JPA | 서울 버스 API 호출, 데이터 가공, 기후동행 필터링 |
| Frontend | React | 지도 표시, 정류장/버스 UI |
| Map | T-Map API (SK) | 지도 렌더링, 마커, 위치, 대중교통 경로 |
| Data | 서울 열린데이터광장 API | 정류장, 도착정보, 노선 정보 |

## 5. 외부 API 및 키 발급

### 5-1. T-Map API (SK Open API)

- **등록 위치**: https://openapi.sk.com
- **발급 키**: App Key (프론트엔드용)
- **사용 기능**: 지도 렌더링, 마커, 현재 위치, 역지오코딩, 대중교통 경로 탐색
- **주의**: 도메인 등록 필요 (개발 시 `localhost` 등록)
- **참고**: 대중교통 경로 API는 Phase 3 (D-03) 구현 시 활용 가능

### 5-2. 서울 열린데이터광장 (공공 버스 API)

- **등록 위치**: https://data.seoul.go.kr
- **발급 키**: 인증키 (서비스 신청 후 발급)
- **사용 API**:

| API 이름 | 용도 |
|----------|------|
| 노선 기본 정보 조회 | 노선 정보 (노선번호, 기점, 종점 등) |
| 정류소 기본 정보 조회 | 정류장 위경도, 이름 |
| 위치 정보로 정류소 목록 조회 | 반경 내 정류장 조회 (F-01 핵심) |
| 버스 도착 정보 조회 | 실시간 도착 예정 시간 (F-02 핵심) |

### 5-3. 기후동행카드 가능 노선 데이터

- **공식 API 없음** → 직접 관리 필요
- **출처**: 서울시 기후동행카드 홈페이지 또는 공지 PDF
- **대응 방안**: DB 테이블로 자체 관리 (`climate_eligible_routes`), 주기적 수동 갱신
- **리스크**: 노선 변경 시 반영 누락 가능 → 갱신 일자 표시 권장

## 7-1. 개발 단계 분리 (Phase Breakdown)

> 전체를 한 번에 개발하지 않고, 단계별로 완성 → 검증 → 다음 단계 진행

### Phase 1: 기반 구축 (Foundation)

**목표**: 개발 환경 설정 + 외부 API 연동 검증
**산출물**: API 연동 테스트 완료, DB 스키마 구성

| 작업 | 내용 |
|------|------|
| BE-01 | Spring Boot 프로젝트 초기 설정 (패키지 구조, 공통 응답 포맷) |
| BE-02 | 서울 열린데이터광장 API 키 발급 및 연동 테스트 |
| BE-03 | `climate_eligible_routes` 테이블 생성 + 초기 데이터 입력 |
| BE-04 | Kakao Map API 키 발급 확인 (프론트 연동 준비) |

**완료 기준**: 서울 버스 API 호출 성공, 기후동행 노선 데이터 DB에 존재

---

### Phase 2: MVP Backend

**목표**: 핵심 3개 API 구현 완료
**산출물**: F-01, F-02, F-03 백엔드 API

| 작업 | 기능 | 내용 |
|------|------|------|
| BE-11 | F-01 | 위치 기반 주변 정류장 조회 API (`GET /api/stops/nearby?lat=&lng=&radius=`) |
| BE-12 | F-02 | 정류장 도착 버스 정보 조회 API (`GET /api/stops/{stationId}/arrivals`) |
| BE-13 | F-03 | 기후동행 가능 여부 필터링 로직 (응답에 `climateEligible: true/false` 포함) |
| BE-14 | 공통 | 서울 버스 API 캐싱 전략 (중복 호출 방지) |

**완료 기준**: Postman 또는 curl로 3개 API 정상 응답 확인

---

### Phase 3: MVP Frontend

**목표**: 지도 기반 UI + 백엔드 연동
**산출물**: 브라우저에서 동작하는 MVP 서비스

| 작업 | 기능 | 내용 |
|------|------|------|
| FE-11 | F-04 | Kakao Map 지도 화면 (React 프로젝트 초기화 포함) |
| FE-12 | F-01 | 현재 위치 감지 + 주변 정류장 마커 표시 |
| FE-13 | F-02 | 정류장 마커 클릭 시 도착 버스 목록 팝업/패널 |
| FE-14 | F-03 | 버스별 기후동행 가능 여부 배지 표시 (`🟢 / 🔴`) |

**완료 기준**: 실제 위치에서 주변 정류장 → 도착 버스 → 기후동행 여부 전체 플로우 동작

---

### Phase 4: 차별화 기능 (P1)

**목표**: 서비스 가치 향상
**선행 조건**: Phase 3 완료

| 작업 | 기능 | 내용 |
|------|------|------|
| D-01 | 필터 | 지도에서 기후동행 가능 정류장만 표시/필터 토글 |
| D-02 | 추천 | 현재 위치 기준 이용 가능한 기후동행 노선 목록 패널 |

---

### Phase 5: 고급 기능 (P2, P3) — 선택

**목표**: 경쟁력 강화 (시간 여유 시 진행)
**선행 조건**: Phase 4 완료

| 작업 | 기능 | 내용 |
|------|------|------|
| D-03 | 경로 탐색 | 기후동행 전용 경로 vs 일반 경로 비교 |
| D-04 | 알림 | 브라우저 Push Notification으로 버스 도착 알림 |

---

## 7-2. 단계별 개발 순서 요약

```
[Phase 1] 기반 구축
    ↓ API 연동 확인 후
[Phase 2] MVP Backend (BE 3개 API)
    ↓ API 완성 후
[Phase 3] MVP Frontend (지도 UI + 연동)
    ↓ MVP 동작 확인 후
[Phase 4] 차별화 기능 (필터, 추천)
    ↓ 여유 시
[Phase 5] 고급 기능 (경로, 알림)
```

> **권장**: Phase 1~3을 먼저 완성해서 실제로 동작하는 MVP를 만든 후, 이후 단계를 순차적으로 추가
