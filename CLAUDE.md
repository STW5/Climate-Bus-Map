## 프로젝트 개요
- 서울 기후동행카드 지원 버스/지하철 경로 안내 서비스
- 모노레포: BE (Spring Boot) + FE (React + Vite)
- Git root: `/Users/stw/Dev/Climate-Bus-Map`

---

## 절대 금지 사항 (NEVER)

### 배포
- `deploy-fe.sh` 절대 실행 금지 — CI/CD가 있음, rsync 직접 배포 금지
- 배포는 항상 `git commit` + `git push origin main`으로만

### 커밋 메시지
- 영어 커밋 메시지 금지 → 반드시 한국어
- "with claude", "by claude" 문구 금지
- "phase3", "Phase 3" 등 phase 번호 금지
- `Co-Authored-By` 줄 금지
- 좋은 예: `feat: 버스 도착 정보 조회 API 구현`
- 나쁜 예: `feat: Phase 3 프론트엔드 구현 with claude`

### 코드 수정
- 읽지 않은 파일 수정 금지 — 반드시 전체 파일을 끝까지 읽고 수정
- 요청하지 않은 리팩토링, 주석 추가, 타입 어노테이션 추가 금지
- 사용하지 않는 import 제거 외에 코드 정리 금지
- 새 파일 생성은 꼭 필요한 경우에만
- 외부 의존성(라이브러리) 추가 전 반드시 사용자에게 먼저 물어볼 것
- 인증/세션/암호화 관련 코드는 직접 구현 금지 — 기존 검증된 라이브러리만 사용

### 보안
- 시크릿(API 키, 비밀번호)을 코드/로그/커밋에 절대 포함 금지
- SQL Injection, XSS, 커맨드 인젝션 등 OWASP Top 10 취약점 도입 금지
- 모든 외부 입력은 반드시 검증 후 사용

## 작업 전 필수 절차

### 1. 파일 읽기
- 수정 전 관련 파일 전체를 끝까지 읽을 것 (부분 읽기 금지)
- 수정 대상 파일뿐 아니라 호출부, 참조 경로, 관련 테스트까지 확인

### 2. 멈추고 물어야 하는 상황 (Stop & Ask)
아래 상황에서는 진행하지 말고 반드시 사용자에게 먼저 확인:
- 여러 파일에 걸친 변경이 예상될 때 → 계획 먼저 제시하고 승인받기
- 인증/보안/암호화 관련 코드를 건드릴 때
- DB 스키마 또는 마이그레이션 변경 시
- 새 외부 라이브러리 추가 시
- 익숙하지 않은 영역에서 가정(assumption)을 해야 할 때

### 3. 변경 최소화 원칙
- 문제를 해결하는 데 필요한 최소한의 줄만 변경
- 요청 범위 밖의 코드는 건드리지 않음
- 기존 코드 스타일을 "더 좋은 스타일"보다 우선시할 것

### 4. 불확실할 때
- 확신이 없으면 추측하지 말고 명시적으로 "불확실합니다"라고 말할 것
- 특히 보안 관련 결정에서 확신이 없으면 반드시 사용자에게 확인
- 가정(assumption)을 했다면 반드시 명시적으로 밝힐 것

### 5. 커밋 전 자가 검증
- AI가 생성한 코드는 커밋 전에 모든 줄을 직접 설명할 수 있어야 함
- 설명할 수 없는 줄이 있으면 커밋하지 말 것
- 변경 후 테스트를 직접 실행하여 통과 확인 후 커밋

## 코딩 규칙

### 크기 제한
- 파일 ≤ 300 LOC, 함수 ≤ 50 LOC, 파라미터 ≤ 5개, 순환 복잡도 ≤ 10
- 초과 시 분리/리팩토링 (단, 요청받은 경우에만)

### 코드 품질
- 의도가 드러나는 이름 사용 (변수명, 함수명에서 의도가 읽혀야 함)
- guard clause 우선 (중첩 if보다 early return)
- 상수 하드코딩 금지 — 심볼/상수로 추출
- 사이드 이펙트(I/O, 네트워크, 전역 상태)는 경계 레이어에 격리
- 특정 예외만 catch, 광범위한 예외 처리 금지
- 구조화된 로그 사용, 민감 데이터 로그 금지

### 옵션 비교
- 결정 전 최소 2가지 구현 방법을 비교 (장단점 한 줄씩)
- 가장 단순한 방법을 선택

## 테스트 규칙
- 새 코드에는 반드시 새 테스트 작성
- 버그 수정 시 regression test 먼저 작성 (실패 확인 후 수정)
- 테스트는 결정적(deterministic)이고 독립적이어야 함
- Mock보다 실제 동작을 검증하는 통합 테스트 선호
- 코드 변경 후 테스트 실행하여 직접 확인

## 아키텍처 규칙

### BE (Spring Boot)
- Hexagonal Architecture 준수
  - `domain/` — 순수 도메인 로직, 외부 의존 없음
  - `web/` — Inbound Adapter (Controller)
  - `external/` — Outbound Adapter (외부 API)
  - `config/` — 설정
- 패키지: `com.stw.climatebusmapbe`
- 새 기능은 반드시 도메인 레이어부터 설계

### FE (React + Vite)
- React 18 + Vite
- T-Map Web SDK v2: 반드시 `useTmapReady` hook으로 동적 로드
- FE 포트: 3000 (BE CORS 설정과 맞춤)
- 환경변수: `VITE_` prefix
